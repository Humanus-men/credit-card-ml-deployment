# Архитектура сервиса и MLOps-концепты

## 1. Монолит vs Микросервисы

В данном проекте выбран **монолитный подход**: один веб-сервис на Flask обрабатывает запросы на предсказание.

**Обоснование**:
- Простота разработки, тестирования и развёртывания в рамках учебного кейса.
- Отсутствие высоких требований к независимому масштабированию компонентов на данном этапе.
- Сервис может быть горизонтально масштабирован за балансировщиком нагрузки, что достаточно для большинства начальных production-сценариев.

При росте нагрузки и усложнении логики можно перейти к микросервисной архитектуре (отдельные сервисы для предобработки, инференса, постобработки) с обменом сообщениями через брокер очередей.

## 2. Брокер сообщений (RabbitMQ) – концепт

В сценарии массовой обработки кредитных заявок возможно применение очередей сообщений (например, RabbitMQ):
- Запросы на скоринг помещаются в очередь.
- Воркеры асинхронно разбирают батчи, выполняют предсказания и возвращают результаты.
- Повышается отказоустойчивость, пропускная способность, появляется возможность приоритезации запросов и гарантированной доставки.

Для учебного проекта реализация RabbitMQ не требуется, достаточно описания концепции.

## 3. Логирование и мониторинг (обзорно)

- Сейчас логи записываются в stdout в формате JSON (timestamp, level, message).
- В production-среде логи собираются агентом (например, Filebeat) и отправляются в ELK-стек (Elasticsearch, Logstash, Kibana) для централизованного хранения, поиска и визуализации.
- Для мониторинга состояния сервиса и алертов могут использоваться Prometheus и Grafana.

## 4. DVC (Data Version Control) и MLflow

- **DVC** – инструмент для версионирования датасетов и артефактов моделей. Позволяет отслеживать, какая версия данных использовалась при обучении конкретной версии модели, и воспроизводить эксперименты.
- **MLflow Tracking** – сервер для записи параметров, метрик и моделей в процессе экспериментов. Упрощает сравнение версий (например, v1 vs v2) и поиск оптимальных конфигураций.

Внедрение этих инструментов в production-пайплайн повышает воспроизводимость и управляемость жизненным циклом модели.

## 5. Бизнес-метрики

Помимо технических метрик (F1, Precision, Recall), для бизнеса важны:

- **Ожидаемые потери (Expected Loss)**: сумма вероятностей дефолта, умноженных на сумму задолженности, для заявок, которые были бы отклонены при заданном пороге. Банк стремится минимизировать ожидаемые потери при фиксированном объёме выдач.
- **Доля одобренных заявок (Approval Rate) при заданном уровне риска**: увеличение количества одобренных кредитов без роста доли плохих долгов свидетельствует о повышении точности модели.

Эти метрики могут быть рассчитаны путём прогона исторических данных через модель и сравнения кумулятивных показателей.

## 6. ONNX и uWSGI + NGINX (дополнительно)

- **ONNX-ML** – открытый формат для представления моделей машинного обучения. Преобразование обученной модели в ONNX позволяет ускорить инференс и упростить развёртывание в различных средах (в том числе без Python).
- **uWSGI + NGINX** – классическая production-связка: uWSGI управляет процессами приложения, а NGINX выступает в роли обратного прокси, обрабатывает статику, терминирует SSL и балансирует нагрузку.

В рамках учебного проекта реализация не требуется, достаточно понимания их назначения.
