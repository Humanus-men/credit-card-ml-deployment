# План A/B-тестирования модели прогнозирования дефолта

## Цель
Сравнить эффективность текущей модели (v1 – Logistic Regression) и новой модели (v2 – Random Forest) в условиях, приближенных к реальным.

## Дизайн теста
- **Контрольная группа (A)**: 50% трафика, обрабатывается моделью v1.
- **Тестовая группа (B)**: 50% трафика, обрабатывается моделью v2.
- **Способ разделения трафика**: случайное распределение на основе идентификатора клиента или сессии (на уровне API-шлюза или балансировщика). В сервисе реализована поддержка выбора версии через HTTP-заголовок `X-Model-Version`.
- **Продолжительность теста**: 2 недели (или до достижения статистической значимости).

## Метрики оценки
- **Основная метрика**: F1-score для класса «дефолт» (положительный класс). F1-score сбалансированно учитывает точность и полноту.
- **Дополнительная метрика**: Recall (полнота) для класса «дефолт», так как пропуск дефолтного клиента обходится дороже, чем ложная тревога.
- **Бизнес-метрика (наблюдаемая)**: доля фактических дефолтов среди одобренных заявок (default rate). Сравнивается в группах A и B при фиксированном уровне одобрений. Меньший уровень дефолта при одинаковой доле одобрений свидетельствует о лучшей модели.

## Статистический анализ
- Для сравнения бинарной метрики (дефолт/не дефолт) используется **z-тест для пропорций**.
- Доверительный интервал для разности долей строится на уровне 95%.
- **Критерий успешности**: статистически значимое (p < 0.05) увеличение F1-score и/или снижение default rate при сохранении (или улучшении) Recall.

## Техническая реализация в сервисе
- Сервис поддерживает две версии модели, загружаемые одновременно.
- Версия выбирается по заголовку `X-Model-Version` (значения `v1` или `v2`). По умолчанию используется `v1`.
- Это позволяет маршрутизировать трафик на уровне обратного прокси (например, nginx с Lua) или балансировщика без внесения изменений в код приложения.
